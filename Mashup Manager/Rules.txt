1. THE ODIN FRAMEWORK (Custom Angular)
--------------------------------------------------------------------------------
Mashups are built with "Odin" — a custom Angular 18 CLI & framework.

  ► Create a new project:   odin new
      → Prompt 1 — "What is the name of the project?"
           Enter the project name (e.g. MyProject — PascalCase or kebab-case)
      → Prompt 2 — "Which view framework do you want to use?"
           Select: Angular
      → Prompt 3 — "Which style library do you want to use?"
           Select: SoHo (Infor Design System)
      → Prompt 4 — "What is the URL of your M3 environment?"
           Enter: https://example.com:8080  (use the real environment URL)
      → Prompt 5 — "Should Git be used for the project?"
           Select: No  (unless version control is required)
      → Prompt 6 — "Should dependencies be installed?"
           Select: Yes
      → After odin new finishes:
           cd <project-name>
           npm install
           odin serve
      → Build: odin build

  NEVER use: ng new / ng serve / ng build / ng generate
  ALWAYS use: odin new / odin serve / odin build / odin g

  UI Library: IDS Enterprise NG (ids-enterprise-ng@18.2.4)
  Import in AppModule: SohoComponentsModule from 'ids-enterprise-ng'

2. COMPONENT LIBRARY (Always consult before generating code)
--------------------------------------------------------------------------------
Location: Mashup Manager/Components Library/

  ids-enterprise-ng/lib/<component>/
    soho-<component>.component.d.ts     ← @Inputs, @Outputs, public API

  scraped_components/<slug>/
    example-index_html/page.html        ← Canonical HTML usage example
    example-<variant>_html/page.html    ← Additional variant examples

  scraped_components/_index.json        ← Master list of all components

  RULE: Before generating ANY IDS component, ALWAYS:
    1. Read the .d.ts for @Input / @Output names
    2. Read at least one example HTML page
    3. Use ONLY documented property names

3. TYPESCRIPT / ANGULAR CODING RULES (Summary)
--------------------------------------------------------------------------------
  ✔ Strict TypeScript — no 'any'
  ✔ OnPush change detection on all components
  ✔ Always unsubscribe (takeUntil(destroy$) pattern)
  ✔ IDS components: access only in ngAfterViewInit (not ngOnInit)
  ✔ One responsibility per component — data fetching in Services
  ✔ trackBy on all *ngFor
  ✔ Type all event handler parameters (check .d.ts for payload types)
  ✔ DataGrid columns defined as SohoDataGridColumn[] typed array
  ✔ HTTP returns Observables, not Promises
  ✔ Error handling with soho-message or soho-toast (never silent failures)

  ✘ Never initialize IDS controls via jQuery in Angular components
  ✘ Never access @ViewChild IDS refs before ngAfterViewInit
  ✘ Never use 'ng' CLI commands — use 'odin' commands

4. PROJECT FOLDER STRUCTURE (Generated by odin new)
--------------------------------------------------------------------------------
  <project>/
    src/
      app/
        app-routing.module.ts
        app.module.ts              ← Import SohoComponentsModule here
        app.component.ts
        app.component.html
        app.component.css
        app.component.spec.ts
      main.ts
      index.html
      styles.css                   ← Global styles (IDS theme CSS imported here)
    public/
      favicon.ico
    .vscode/
      extensions.json
      launch.json
      tasks.json
    angular.json
    package.json
    tsconfig.json
    tsconfig.app.json
    tsconfig.spec.json
    README.md

  NOTE: odin generates .css files (not .scss). Do not rename them.
  NOTE: odin generates a public/ folder (not assets/) for static files.
  NOTE: No components/, services/, or models/ folders are generated — create
        them manually under src/app/ as you build features.

6. HOW TO USE CURSOR WITH THIS WORKSPACE
--------------------------------------------------------------------------------
  1. Open this workspace in Cursor (c:\Users\A-0813\Documents\Cursor\)
  2. All rules in .cursor/rules/ are automatically applied to AI responses
  3. To create a mashup, open Cursor Chat and use a prompt from:
       .cursor/rules/mashup-prompt-templates.mdc
  4. New projects go in:  Mashup Manager/Projects/<project-name>/
  5. The AI will automatically consult the Components Library when asked
     to generate any IDS-based component

     will also read matching patterns from Mashup Manager/Projects Examples/
     to ensure the generated code follows the real-world conventions used
     in this workspace (standalone components, IMIRequest pattern, Store<T>,
     GDIS services, path aliases, etc.)
